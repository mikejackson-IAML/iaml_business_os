**Workflow settings:**
- name: "WF5: Publishing & First Comment"
- timezone: "America/Chicago"
- executionTimeout: 600 (10 minutes, needed for Wait node)
- tags: ["linkedin-content-engine"]

**Node architecture (~22-25 nodes):**

**Main flow:**
1. **Schedule Trigger** - Cron: `0 14 * * 2-5` (Tue-Fri 8 AM CST = 14:00 UTC). typeVersion 1.2.
2. **Code: Build Today's Date** - Use `$now.toISODate()` (respects workflow timezone). Generate a `run_id` UUID and `started_at` timestamp.
3. **HTTP Request: Log Run Start** - POST to `workflow_runs` table. workflow_name: "wf5-publishing-first-comment". status: "running". Use Supabase REST credential `Dy6aCSbL5Tup4TnE` with Accept-Profile/Content-Profile: linkedin_engine headers. Match exact header pattern from WF4.
4. **HTTP Request: Fetch Calendar Slot** - GET `content_calendar?post_date=eq.{today}&select=*`. Schema-scoped headers. Filter for slots that have a non-null `post_id`.
5. **IF: Has Calendar Slot with Post** - Check if response array is non-empty AND first item has a `post_id`. If no slot or no post_id, go to "Log Skip" node.
6. **HTTP Request: Fetch Post Details** - GET `posts?id=eq.{post_id}&select=*`. Fetch the full post record.
7. **IF: Post Status is Approved** - Check `$json[0].status === 'approved'`. If not approved, go to "Log Skip" node.
8. **HTTP Request: Set Status to Scheduled** - PATCH `posts?id=eq.{post_id}` with body `{"status": "scheduled"}`. This marks intent to publish.
9. **LinkedIn Node: Publish Post** - Use `n8n-nodes-base.linkedIn` typeVersion 1. Set person field to `urn:li:person:PLACEHOLDER_PERSON_URN` (user replaces after OAuth2 setup). Set text to the post's `full_text`. Credential: linkedInOAuth2Api with placeholder id `PLACEHOLDER_LINKEDIN_CREDENTIAL_ID`.
10. **Code: Extract Post URN** - Extract URN from LinkedIn node response. Check `$json.id`, `$json.urn`, and `$json['x-restli-id']` fields. Throw error if no URN found. Construct LinkedIn URL: `https://www.linkedin.com/feed/update/{urn}`. Return `linkedin_post_urn`, `linkedin_url`, `published_at`.
11. **HTTP Request: Update Post Published** - PATCH `posts?id=eq.{post_id}` with body `{"status": "published", "linkedin_post_id": "{urn}", "published_at": "{timestamp}"}`.
12. **HTTP Request: Update Calendar Published** - PATCH `content_calendar?id=eq.{calendar_id}` with body `{"status": "published"}`.
13. **Wait Node** - 45 seconds. resume: "timeInterval", interval: 45, unit: "seconds". typeVersion 1.1.
14. **HTTP Request: Post First Comment** - POST to `https://api.linkedin.com/rest/socialActions/{encodedUrn}/comments`. Use `predefinedCredentialType: "linkedInOAuth2Api"` for auth. Headers: Content-Type application/json, X-Restli-Protocol-Version 2.0.0, LinkedIn-Version 202402. Body: `{"actor": "urn:li:person:PLACEHOLDER_PERSON_URN", "object": "{postUrn}", "message": {"text": "{first_comment_text}"}}`. URL-encode the URN with `encodeURIComponent()`.
15. **HTTP Request: Slack Success Notification** - POST to `https://hooks.slack.com/services/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`. Rich Slack blocks format with: post title (hook_text), series, pillar, hook category + variation, character count of full_text, LinkedIn link. Use green checkmark emoji in the message.
16. **HTTP Request: Log Run Complete** - PATCH `workflow_runs` to update status to "completed", items_processed to 1, ended_at timestamp.

**Skip flow (no post to publish):**
17. **Code: Log Skip** - When no calendar slot, no post_id, or post not approved. Set skip reason.
18. **HTTP Request: Log Skip Run** - PATCH `workflow_runs` status to "completed", items_processed to 0, metadata includes skip_reason.

**Retry flow (LinkedIn publish failure):**
19. **Code: Check Retry Count** - If publish fails, check retry_count. If < 1, set should_retry=true and increment retry_count.
20. **IF: Should Retry** - If yes, connect to a Wait 5 min node then back to the LinkedIn publish node (or a duplicate publish node). If no, connect to revert flow.
21. **Wait: 5 Min Retry Delay** - 300 seconds wait before retry.
22. **HTTP Request: Revert to Approved** - PATCH `posts?id=eq.{post_id}` with `{"status": "approved"}`. Only reached after retry exhausted.
23. **HTTP Request: Slack Error Alert** - POST to same Slack webhook with error details (red X emoji, error message, post title).

**Comment failure flow:**
- Connect the First Comment HTTP Request error output to a retry-once branch. If comment retry also fails, send Slack alert but do NOT change post status (post stays published).

**Canary error handling (global):**
24. **Error Trigger** - n8n-nodes-base.errorTrigger.
25. **HTTP Request: Log Error** - POST to `workflow_runs` with status "failed", error details in metadata. Same Supabase REST pattern as all other workflows.
26. **HTTP Request: Slack Error Alert (Canary)** - POST to Slack with error details.

**Node positioning:** Follow WF4 layout conventions. Main flow left-to-right. Error branch below main flow offset by +300 Y. Skip branch at +200 Y offset.

**Connections array:** Wire all nodes correctly with main and error outputs. The LinkedIn node error output connects to the retry check. The comment HTTP Request error output connects to a comment retry branch.

**IMPORTANT PLACEHOLDERS:**
- `PLACEHOLDER_PERSON_URN` - User replaces with `urn:li:person:{their_sub_id}` after OAuth2 setup
- `PLACEHOLDER_LINKEDIN_CREDENTIAL_ID` - User replaces with the credential ID from n8n after creating the LinkedIn OAuth2 credential

These placeholders are intentional and documented. The user must complete LinkedIn OAuth2 setup before importing this workflow.

**Pattern references:**
- Supabase REST headers: Exactly match WF4's httpHeaderAuth with Dy6aCSbL5Tup4TnE
- Code nodes: Match WF4's jsCode pattern for UUID generation and data extraction
- Error trigger: Match WF1's canary error handling pattern
- All HTTP Request nodes: typeVersion 4.2, timeout 10000
