  <action>
    **1. Expand mutation functions** in `dashboard/src/lib/api/linkedin-content-mutations.ts`:

    Add these functions following the exact same pattern as `updateTopicStatus` (dot notation, `as never` assertion, `as PostDb` return):

    a) `updateDraftStatus(id: string, status: 'approved' | 'rejected' | 'draft'): Promise<PostDb>`
       - Update `linkedin_engine.posts` status
       - When approving: also set `scheduled_for` if the post has a calendar slot

    b) `selectHookVariation(id: string, variation: 'A' | 'B' | 'C', hookVariations: { text: string; category: string; variation: string }[]): Promise<PostDb>`
       - Find the selected hook in the hookVariations array
       - Update `hook_text`, `hook_category`, `hook_variation` on the post row
       - Throw if variation not found in array

    c) `updateDraftText(id: string, updates: { full_text?: string; first_comment_text?: string; hook_text?: string }): Promise<PostDb>`
       - Partial update: only set fields that are provided
       - Used for inline editing of post body, first comment, or hook text

    d) `triggerRegeneration(id: string, instructions: string, regenerateScope: 'hooks' | 'body' | 'full'): Promise<{ success: boolean; message: string }>`
       - First: PATCH the post to set `generation_status: 'regenerating'` and `generation_instructions: instructions`
       - Then: fire-and-forget POST to `https://n8n.realtyamp.ai/webhook/linkedin-content-generate` with body `{ topic_id: post.topic_id, regenerate: true, scope: regenerateScope, instructions }`
       - The n8n webhook handles the actual regeneration (updates existing post row, not insert new)
       - Use try/catch: if webhook call fails, still return success (post is marked regenerating, user can retry)
       - Need to fetch the post first to get `topic_id`

    e) `assignCalendarSlot(postId: string, calendarSlotId: string): Promise<ContentCalendarDb>`
       - PATCH `linkedin_engine.content_calendar` to set `post_id`, `status: 'generated'`
       - Used for manual calendar slot reassignment

    Import `PostDb` and `ContentCalendarDb` types from `linkedin-content-queries.ts`.

    **2. Create 5 API routes** following the exact pattern from `topics/[id]/status/route.ts`:

    Each route:
    - Imports from `linkedin-content-mutations`
    - Has `RouteContext` interface with `params: Promise<{ id: string }>`
    - UUID validates the id param
    - Parses JSON body
    - Validates required fields
    - Calls mutation function
    - Returns JSON response
    - Wraps in try/catch with consistent error format

    a) `drafts/[id]/status/route.ts` -- PATCH
       - Body: `{ status: 'approved' | 'rejected' | 'draft' }`
       - Calls `updateDraftStatus`

    b) `drafts/[id]/hook/route.ts` -- PATCH
       - Body: `{ variation: 'A' | 'B' | 'C', hook_variations: [{text, category, variation}] }`
       - Calls `selectHookVariation`

    c) `drafts/[id]/edit/route.ts` -- PATCH
       - Body: `{ full_text?: string, first_comment_text?: string, hook_text?: string }`
       - At least one field required
       - Calls `updateDraftText`

    d) `drafts/[id]/regenerate/route.ts` -- POST
       - Body: `{ instructions: string, scope: 'hooks' | 'body' | 'full' }`
       - Instructions required, scope defaults to 'full'
       - Calls `triggerRegeneration`

    e) `drafts/[id]/calendar/route.ts` -- PATCH
       - Body: `{ calendar_slot_id: string }`
       - UUID validates both the draft id and calendar_slot_id
       - Calls `assignCalendarSlot`

    **3. Modify existing topic status route** `topics/[id]/status/route.ts`:

    After successfully updating status to 'approved', add a fire-and-forget webhook call:
    ```typescript
    if (status === 'approved') {
      fetch('https://n8n.realtyamp.ai/webhook/linkedin-content-generate', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ topic_id: id }),
      }).catch((err) => {
        console.error('Failed to trigger WF4:', err);
      });
    }
    ```
    This is non-blocking: the topic is approved even if the webhook fails. User can manually trigger regeneration from the Drafts tab.
