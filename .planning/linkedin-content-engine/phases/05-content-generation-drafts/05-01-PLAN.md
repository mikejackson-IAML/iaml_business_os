    Build the WF4 n8n workflow JSON following the exact pattern from WF3 (`wf3-topic-scoring-engine.json`).

    **Trigger:** Webhook node (typeVersion 2), POST method, path `linkedin-content-generate`, `responseMode: "immediately"`. This means the webhook responds with 200 + run_id immediately and generation runs async. The webhook body contains `{ topic_id: "<uuid>" }`.

    **Node sequence (approximately 14-16 nodes):**

    1. **Webhook Trigger** (n8n-nodes-base.webhook, typeVersion 2)
       - `httpMethod: "POST"`, `path: "linkedin-content-generate"`, `responseMode: "immediately"`

    2. **Validate Input & Generate Run ID** (Code node)
       - Extract `topic_id` from webhook body
       - Generate UUID for `run_id`
       - Calculate `week_of` (Monday of current week) using same pattern as WF3
       - Validate topic_id is present

    3. **Log Run Start** (HTTP Request POST to `workflow_runs`)
       - Same pattern as WF3: `workflow_name: "wf4-content-generation-pipeline"`, status: "running"
       - Use credential `Dy6aCSbL5Tup4TnE`, headers: `Content-Profile: linkedin_engine`, `Accept-Profile: linkedin_engine`
       - Base URL: `https://mnkuffgxemfyitcjnjdc.supabase.co/rest/v1/`

    4. **Update Topic Generation Status** (HTTP Request PATCH to `topic_recommendations`)
       - Set a metadata flag or rely on the post creation to signal generation is in progress
       - Actually: skip this node. The generation_status is on the posts table, not topic_recommendations.

    5. **Fetch Approved Topic** (HTTP Request GET to `topic_recommendations`)
       - `?id=eq.{topic_id}&select=*`
       - Must return the topic with angle, recommended_format, recommended_series, key_data_points, source_signal_ids

    6. **Fetch Source Signals** (HTTP Request GET to `research_signals`)
       - Use `source_signal_ids` from the topic to fetch related research signals
       - `?id=in.({comma_separated_ids})&select=title,body_text,source,keywords,topic_category`
       - Truncate `body_text` to 500 chars per signal (same as WF3)

    7. **Fetch Top Hooks** (HTTP Request GET to `hooks`)
       - `?status=eq.active&order=score.desc&limit=5&select=hook_text,hook_category`
       - Gets top 5 hooks from library for context

    8. **Find Calendar Slot** (HTTP Request GET to `content_calendar`)
       - Find the next open calendar slot for assignment
       - `?status=eq.open&post_date=gte.{today}&order=post_date.asc&limit=1&select=*`
       - This gives us the series and pillar for framing

    9. **Assemble Context Package** (Code node)
       - Build the full Claude prompt using PROMPT.md template structure
       - Include: topic_title, angle, format, series (from topic OR calendar slot), pillar (from calendar slot), key_data_points, research context (truncated signals), product phase ("Phase 1 (Current): The Ingestion Engine"), top hooks
       - Pillar-specific framing instructions per PROMPT.md
       - AEO terms guidance
       - Output format: JSON with `hook_a`, `hook_b`, `hook_c` (each with `text` and `category`), `full_text`, `first_comment_text`, `hashtags`, `series`, `pillar`
       - Include strict "respond with ONLY valid JSON" instruction

    10. **Claude: Generate Content** (HTTP Request POST to `https://api.anthropic.com/v1/messages`)
        - Same auth pattern as WF3: `predefinedCredentialType: "httpHeaderAuth"`, credential `anthropic-api`
        - Headers: `anthropic-version: 2023-06-01`, `Content-Type: application/json`
        - Body: `model: "claude-sonnet-4-20250514"`, `max_tokens: 4000`, messages array with system + user prompt
        - Timeout: 120000ms (content generation is slower than scoring)

    11. **Parse Generation Response** (Code node)
        - Use the defensive parsing pattern from WF3: strip markdown fences, trim, try/catch JSON.parse
        - Extract: hook_a, hook_b, hook_c, full_text, first_comment_text, hashtags, series, pillar
        - Build `hook_variations` JSONB array: `[{text: hook_a.text, category: "data", variation: "A"}, {text: hook_b.text, category: "contrarian", variation: "B"}, {text: hook_c.text, category: "observation", variation: "C"}]`
        - Default selected hook to Hook A (data variation)
        - On parse failure: set `generation_status: "failed"`, include raw response in error_message

    12. **Insert Post Draft** (HTTP Request POST to `posts`)
        - Insert single row with:
          - `topic_id`, `hook_text` (Hook A text), `hook_category` ("data"), `hook_variation` ("A")
          - `hook_variations` (JSONB array of all 3), `full_text`, `first_comment_text`
          - `format` (from topic or generation), `series`, `pillar`, `hashtags`
          - `status: "draft"`, `generation_status: "completed"`
        - Use `Prefer: return=representation` to get the inserted row back with its `id`

    13. **Assign Calendar Slot** (HTTP Request PATCH to `content_calendar`)
        - PATCH the calendar slot found in node 8: set `post_id` to the new post ID, `topic_id`, `status: "generated"`
        - If no open slot was found, skip this node (use an IF node or handle in the code node)

    14. **Slack Notification** (HTTP Request POST)
        - URL: `https://hooks.slack.com/services/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`
        - Message: "Content draft generated for: *{topic_title}*\nSeries: {series}\nPillar: {pillar}\nHooks: 3 variations ready\n<dashboard_url|Review in Dashboard>"

    15. **Log Run Complete** (HTTP Request PATCH to `workflow_runs`)
        - Update run status to "completed", set `completed_at`, `items_processed: 1`

    **Error handling branch:**
    16. **Error Trigger** (n8n-nodes-base.errorTrigger) connected to:
        - **Retry Logic** (Code node): Check if this is the first attempt (no retry flag in metadata). If first attempt, set retry flag and route back to Claude call. If second attempt, proceed to error logging.
        - **Log Error** (HTTP Request PATCH to workflow_runs): Set status "failed", error_message
        - **Update Post Status** (HTTP Request PATCH to posts): Set `generation_status: "failed"` if post row exists
        - **Slack Error Alert** (HTTP Request POST): "Content generation FAILED for topic: {topic_title}. Error: {message}. Manual retry available in dashboard."

    **Connection wiring:** Each node connects to the next sequentially. Error trigger connects to retry/error branch. Use the same `connections` format as WF3.
