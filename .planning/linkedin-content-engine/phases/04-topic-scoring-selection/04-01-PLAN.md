<objective>
Build WF3 (Topic Scoring Engine) as a complete n8n workflow JSON file. This workflow runs Monday 5 AM CST, fetches unprocessed research signals for the current week, uses a two-pass Claude approach (Pass 1: cluster signals into topics, Pass 2: score each topic across 5 dimensions), inserts scored topics into `topic_recommendations`, marks signals as processed, and logs the run.

Purpose: This is the intelligence layer that transforms raw research signals (from WF1 and WF2) into ranked, scored topic recommendations visible in the dashboard. Without scoring, the content generation pipeline has no prioritized topics to work with.

Output: One importable n8n workflow JSON file + README documentation + n8n-brain pattern registration.
</objective>

### Schedule Trigger
- Node ID: `schedule-monday-5am-cst`
- Type: `n8n-nodes-base.scheduleTrigger`, typeVersion 1.2
- Cron: Monday at 5:00 AM CST (11:00 UTC). Use cron rule `0 11 * * 1`

### Calculate Week + Start Run Log
- Node ID: `calc-week-and-start-run`
- Type: `n8n-nodes-base.code`, typeVersion 2
- Code: Calculate `signal_week` (Monday of current week in YYYY-MM-DD format) using the same getMondayOfWeek pattern from WF1. Also generate a `run_id` (UUID v4) and `started_at` (ISO timestamp). Return `{ signal_week, week_of: signal_week, run_id, started_at }`.

### Log Run Start
- Node ID: `log-run-start`
- Type: `n8n-nodes-base.httpRequest`
- Method: POST to `https://htmnsoqkwtfshavqxlrm.supabase.co/rest/v1/workflow_runs`
- Headers: `Content-Profile: linkedin_engine`, `Prefer: return=minimal`
- Credential: httpHeaderAuth `Dy6aCSbL5Tup4TnE`
- Body: `{ "workflow_name": "wf3-topic-scoring-engine", "status": "running", "started_at": "{{ $json.started_at }}", "items_processed": 0 }`

### Fetch Unprocessed Signals
- Node ID: `fetch-unprocessed-signals`
- Type: `n8n-nodes-base.httpRequest`
- Method: GET to `https://htmnsoqkwtfshavqxlrm.supabase.co/rest/v1/research_signals?processed=eq.false&signal_week=eq.{{ $json.signal_week }}&select=id,source,source_url,title,body_text,author,platform_engagement,keywords,topic_category,sentiment,collected_date`
- Headers: `Accept-Profile: linkedin_engine`
- Credential: httpHeaderAuth `Dy6aCSbL5Tup4TnE`

### Check Has Signals
- Node ID: `check-has-signals`
- Type: `n8n-nodes-base.if`
- Condition: Check if the response array length > 0. Use the expression `{{ $json.length > 0 }}` or check the HTTP response body array length.
- True branch -> continue to clustering
- False branch -> `log-empty-run` (log workflow_runs with items_processed=0, status=completed, then stop)

### Log Empty Run (false branch)
- Node ID: `log-empty-run`
- Type: `n8n-nodes-base.httpRequest`
- Method: POST to `https://htmnsoqkwtfshavqxlrm.supabase.co/rest/v1/workflow_runs`
- Body: `{ "workflow_name": "wf3-topic-scoring-engine", "status": "completed", "items_processed": 0, "metadata": { "reason": "no_unprocessed_signals" } }`
- Headers: `Content-Profile: linkedin_engine`, `Prefer: return=minimal`

### Prepare Signals for Clustering
- Node ID: `prepare-signals-for-clustering`
- Type: `n8n-nodes-base.code`, typeVersion 2
- Code: Take the raw signals array from the HTTP response. Store the full signal IDs list separately (for later marking as processed). Truncate each signal's `body_text` to 500 characters for the Claude clustering call. Build a JSON string of all signals with fields: id, source, title, body_text (truncated), keywords, topic_category, platform_engagement. Store both `signal_ids_to_process` (full array of UUIDs) and `signals_for_claude` (truncated JSON string).

### Claude Pass 1: Cluster Signals into Topics
- Node ID: `claude-cluster-topics`
- Type: `n8n-nodes-base.httpRequest`
- Method: POST to `https://api.anthropic.com/v1/messages`
- Headers: `x-api-key: {{ $credentials.httpHeaderAuth.value }}` (use anthropic-api credential), `anthropic-version: 2023-06-01`, `Content-Type: application/json`
- Credential: httpHeaderAuth `anthropic-api`
- Body (JSON): Use `claude-sonnet-4-20250514` model, max_tokens 4000. System prompt: "You are a content strategist analyzing research signals for a LinkedIn content engine targeting HR professionals interested in AI." User message includes the clustering prompt from RESEARCH.md with the truncated signals JSON. Request response as a JSON object with a `topics` array. Each topic has: `topic_title` (string), `angle` (string), `signal_ids` (string array of UUIDs), `key_data_points` (string array of 2-3 facts).

  IMPORTANT: First try using `output_config.format` with `json_schema` for structured output. The schema should enforce the topics array structure. If the model does not support `output_config`, fall back to prompt-based JSON by including "Respond with ONLY a valid JSON object, no markdown fences or commentary" in the user message. Since WF1/WF2 used prompt-based JSON successfully, the fallback is proven to work.

  The clustering prompt should instruct Claude to:
  - Group signals into 6-10 distinct topic clusters
  - Each topic represents a single coherent theme for one LinkedIn post
  - A signal can belong to only one topic
  - Topics should be specific (not "AI in HR" -- too broad)
  - Prioritize topics where multiple signals converge
  - Include source signal IDs for traceability
  - Include 2-3 key data points from the constituent signals

### Parse Clustering Response
- Node ID: `parse-clustering-response`
- Type: `n8n-nodes-base.code`, typeVersion 2
- Code: Parse the Claude response. Extract the JSON from `content[0].text`. Parse the `topics` array. For each topic, attach the full signal data (not truncated) for the signals in that topic's `signal_ids` array -- look up from the original fetched signals. Output one item per topic, each containing: `topic_title`, `angle`, `signal_ids`, `key_data_points`, `signals` (full signal objects for this topic's signals), `week_of`, `signal_ids_to_process` (carried forward for final processing step).

### SplitInBatches: Score Each Topic
- Node ID: `split-topics-batch`
- Type: `n8n-nodes-base.splitInBatches`, typeVersion 3
- Batch size: 1 (process one topic at a time to avoid Claude rate limits)

### Claude Pass 2: Score Topic
- Node ID: `claude-score-topic`
- Type: `n8n-nodes-base.httpRequest`
- Method: POST to `https://api.anthropic.com/v1/messages`
- Headers: Same as Pass 1 (anthropic-api credential, anthropic-version 2023-06-01)
- Body (JSON): Use `claude-sonnet-4-20250514` model, max_tokens 1500. User message includes the full scoring prompt from RESEARCH.md with:
  - Topic title and angle
  - The topic's constituent signals (with engagement data where available)
  - Key data points
  - Today's date (for freshness calculation)
  - The complete 5-dimension scoring rubric with exact point values
  - AEO bonus instructions: +3 to positioning_score if topic naturally allows "Agentic RAG", "Compliance Guardrails", "Multi-Agent Orchestration", "HR Agentic Systems"
  - Instructions to recommend format (text/carousel/data_graphic), series (not_being_told/compliance_radar/ask_ai_guy/flex), and hook_suggestion
  - Scoring rationale

  Request JSON response with fields: engagement_score, freshness_score, gap_score, positioning_score, format_score, recommended_format, recommended_series, hook_suggestion, scoring_rationale.

  IMPORTANT for engagement scoring: Instruct Claude that RSS signals have no platform_engagement data. Score engagement based on multi-source coverage (+5 for 3+ sources covering same topic) rather than individual upvote/like counts. Only use upvote/like thresholds for Reddit and LinkedIn signals that have platform_engagement data.

  Use `output_config.format` with json_schema if supported, otherwise prompt-based JSON (same fallback as Pass 1).

### Parse Score + Calculate Total
- Node ID: `parse-score-and-total`
- Type: `n8n-nodes-base.code`, typeVersion 2
- Code: Parse Claude scoring response JSON. Extract the 5 dimension scores. Validate ranges: engagement 0-25, freshness 0-25, gap 0-20, positioning 0-18 (15 base + 3 AEO), format 0-15. Clamp any out-of-range values. Calculate `total_score` as the sum of all 5 dimension scores (can exceed 100 due to AEO bonus -- this is by design). Build the insert payload for topic_recommendations:
  ```
  {
    week_of: <signal_week>,
    topic_title: <from clustering>,
    angle: <from clustering>,
    total_score: <sum>,
    engagement_score: <from Claude>,
    freshness_score: <from Claude>,
    gap_score: <from Claude>,
    positioning_score: <from Claude>,
    format_score: <from Claude>,
    recommended_format: <from Claude>,
    recommended_series: <from Claude>,
    hook_suggestion: <from Claude>,
    key_data_points: <from clustering, as JSONB>,
    source_signal_ids: <signal_ids array from clustering>,
    status: "pending"
  }
  ```

- Method: POST to `https://htmnsoqkwtfshavqxlrm.supabase.co/rest/v1/topic_recommendations`
- Headers: `Content-Profile: linkedin_engine`, `Prefer: return=representation`
- Credential: httpHeaderAuth `Dy6aCSbL5Tup4TnE`
- Body: The topic recommendation payload from the previous Code node.

### Mark Signals Processed (after SplitInBatches done)
- Node ID: `mark-signals-processed`
- Type: `n8n-nodes-base.httpRequest`
- Method: PATCH to `https://htmnsoqkwtfshavqxlrm.supabase.co/rest/v1/research_signals?id=in.({{ $json.signal_ids_to_process.join(',') }})`
- Headers: `Content-Profile: linkedin_engine`, `Prefer: return=minimal`
- Credential: httpHeaderAuth `Dy6aCSbL5Tup4TnE`
- Body: `{ "processed": true }`
- CRITICAL: Use the `signal_ids_to_process` array captured at fetch time, NOT a blanket `processed=eq.false` filter. This prevents race conditions if WF1/WF2 insert new signals during WF3 execution.

### Log Run Complete
