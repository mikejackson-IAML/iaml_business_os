# Milestone v1.1: Analytics & Insights

**Status:** SHIPPED 2026-01-22
**Phases:** 6-8
**Total Plans:** 10

## Overview

This milestone added analytics and insights capabilities to the existing Faculty Scheduler system. Build order prioritized data collection first (response tracking), then historical data (instructor history), and finally alerting (dashboard alerts).

**Build Order Rationale:**
1. Response Tracking first - Needed to identify "viewed but not claimed" instructors
2. Instructor History second - Builds on existing data, informs assignment decisions
3. Dashboard Alerts last - Uses response tracking data to power alerts

## Phases

### Phase 6: Response Tracking

**Goal**: Track when instructors click their magic links so dashboard shows viewed vs not-viewed status
**Depends on**: v1.0 (existing notification and magic token system)
**Plans**: 3 plans

Plans:
- [x] 06-01-PLAN.md — Database migration: Add viewed_at to notifications, update views
- [x] 06-02-PLAN.md — Token validation: Record view on magic link click
- [x] 06-03-PLAN.md — Dashboard: Viewed badge with tooltip, Not Viewed first sort

**Success Criteria:**
1. When instructor clicks magic link, viewed_at timestamp is recorded in notification record
2. Dashboard shows "Viewed" badge on instructors who have opened their notification
3. "Not Responded" list distinguishes between "Viewed, No Claim" and "Not Viewed"

---

### Phase 7: Instructor History

**Goal**: Store and display instructor teaching history for better assignment decisions
**Depends on**: Phase 6 (uses similar dashboard patterns)
**Plans**: 3 plans

Plans:
- [x] 07-01-PLAN.md — Database migration: Create teaching_history table, populate from claims
- [x] 07-02-PLAN.md — Dashboard query and types for instructor history
- [x] 07-03-PLAN.md — Dashboard UI: History display in instructor details and assign modal

**Success Criteria:**
1. Teaching history table stores past program assignments (instructor_id, program_id, dates, completed)
2. Admin can view instructor's teaching history in dashboard
3. Assign modal shows instructor's past programs when selecting

---

### Phase 8: Dashboard Alerts

**Goal**: Surface alerts for programs at risk and unresponsive VIP instructors
**Depends on**: Phase 6 (response tracking needed for VIP non-response alerts)
**Plans**: 4 plans

Plans:
- [x] 08-01-PLAN.md — Database migration: Create alerts table with configurable thresholds
- [x] 08-02-PLAN.md — Dashboard query and types for alerts
- [x] 08-03-PLAN.md — Dashboard UI: Alert badge, alert list, dismiss action
- [x] 08-04-PLAN.md — Documentation: Alert refresh workflow specification

**Success Criteria:**
1. Alert badge shows count of active alerts in dashboard header
2. "Approaching Tier End" alerts fire 24h before tier window closes (configurable)
3. "VIP Non-Response" alerts fire when VIP hasn't viewed after N days (configurable)
4. Alerts can be dismissed/acknowledged from dashboard

## Milestone Summary

**Key Decisions:**

- Decision: Record view on magic link click even for expired tokens (captures engagement intent)
- Decision: Trigger-based history population (automatic, no manual intervention needed)
- Decision: On-demand alert refresh on dashboard load (no separate scheduled job needed)
- Decision: Configurable thresholds via n8n_brain.preferences (same pattern as other Business OS components)

**Issues Resolved:**

- Determining when to record view (settled on first click only, via viewed_at IS NULL check)
- Alert auto-resolution when VIP views notification (built into refresh_alerts function)
- Optimistic UI for alert dismissal with undo window (10 seconds)

**Issues Deferred:**

- Periodic n8n workflow for alert refresh (documented as optional, dashboard query sufficient)
- Response time analytics beyond simple viewed/not-viewed

**Technical Debt Incurred:**

- None identified during milestone audit
