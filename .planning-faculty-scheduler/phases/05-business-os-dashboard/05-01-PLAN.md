# 05-01: Supabase Dashboard Views

## Objective
Create enhanced Supabase views that aggregate all dashboard data requirements: notification counts, response rates, instructor assignments, and "not responded" tracking.

## Context
The existing faculty_scheduler schema has basic tables but lacks aggregated views for dashboard display:
- Notification counts per program
- Response counts (instructors who claimed vs notified)
- Assigned instructor names
- "Not responded" instructor tracking

## Tasks

<task id="1">
Create migration file `supabase/migrations/20260122_faculty_scheduler_phase5_dashboard.sql` with header comment explaining Phase 5 dashboard views.
</task>

<task id="2">
Create `faculty_scheduler.dashboard_recruitment_pipeline` view that provides all columns needed for the dashboard table:
- Program ID, name, program_type, city, state, start_date
- Current status (tier_0/tier_1/tier_2/filled/completed/draft)
- released_at, tier_0_ends_at, tier_1_ends_at
- days_remaining: calculated from current tier end date
- total_blocks, open_blocks, filled_blocks: aggregated from program_blocks
- notified_count: COUNT of distinct instructors with tier_release notifications for this program
- responded_count: COUNT of distinct instructors with claims for blocks in this program
- assigned_instructor_name: For claimed programs, first assigned instructor name
- assigned_instructor_id: UUID of assigned instructor
- last_activity_at: MAX of (program updated_at, latest claim.created_at, latest notification.created_at)
- tier_display: CASE mapping status to friendly names ('Tier 0 (VIP)', 'Tier 1 (Local)', 'Open', 'Filled', 'Completed', 'Draft')

The view should use LEFT JOINs to include programs with no notifications or claims.
</task>

<task id="3">
Create `faculty_scheduler.not_responded_instructors` view showing instructors who were notified but haven't responded:
- Columns: instructor_id, full_name, email, firm_state, tier_designation, scheduled_program_id, program_name, program_city, program_state, notified_at, tier_when_notified
- Logic: SELECT from notifications WHERE notification_type='tier_release' AND NOT EXISTS (claim for this instructor + this program)
- Only include active programs (status IN ('tier_0', 'tier_1', 'tier_2'))
- ORDER BY: notified_at DESC
</task>

<task id="4">
Create `faculty_scheduler.dashboard_summary_stats` view for summary cards (single row):
- total_programs: COUNT of all non-draft programs
- awaiting_tier_0: COUNT WHERE status='tier_0'
- awaiting_tier_1: COUNT WHERE status='tier_1'
- open_programs: COUNT WHERE status='tier_2'
- filled_programs: COUNT WHERE status IN ('filled', 'claimed', 'confirmed')
- draft_programs: COUNT WHERE status='draft'
- programs_needing_attention: COUNT WHERE days_remaining < 2 AND status IN ('tier_0', 'tier_1', 'tier_2')
- total_notified: Total distinct instructor notifications across all programs
- total_responded: Total distinct instructors with claims across all programs
- response_rate: (total_responded::numeric / NULLIF(total_notified, 0) * 100) or 0
</task>

<task id="5">
Create `faculty_scheduler.assign_instructor(p_block_id UUID, p_instructor_id UUID)` function for manual assignment:
- Returns: TABLE (success BOOLEAN, error_message TEXT, claim_id UUID)
- Logic:
  1. Verify block exists and status='open'
  2. Verify instructor exists in faculty table and active=true
  3. Create claim record with status='confirmed' (admin assignment)
  4. Update program_blocks: instructor_id, claimed_at=NOW(), status='claimed'
  5. Check if all blocks filled -> update scheduled_program status to 'filled'
- Does NOT check tier eligibility (admin override capability)
</task>

<task id="6">
Create `faculty_scheduler.override_claim(p_claim_id UUID, p_reason TEXT)` function for removing claims:
- Returns: TABLE (success BOOLEAN, error_message TEXT, block_id UUID, program_id UUID)
- Logic:
  1. Find claim with status='confirmed' or 'claimed'
  2. Update claim: status='cancelled', cancelled_at=NOW(), cancelled_reason=p_reason
  3. Re-open the block: instructor_id=NULL, claimed_at=NULL, status='open'
  4. Recalculate program status (may need to go back to current tier)
  5. Return block_id and program_id for re-release notification
</task>
