# 06-02: Faculty Portal - Record View on Magic Link Validation

## Objective
Modify the `validate_magic_token()` RPC function to record the notification view when an instructor clicks their magic link. The view should be recorded on first click only, even if the token is expired or invalid.

## Context
The `validate_magic_token()` function in Phase 2 migration already updates token usage stats (`last_used_at`, `use_count`). We need to also call the `record_notification_view()` helper function created in 06-01 to track when instructors first see their notification.

Per context decisions:
- Record timestamp on first click only (handled by helper function)
- Record view even if token is expired (instructor saw notification)
- No side effects - viewing does not change notification status

The faculty portal calls `validate_magic_token` via RPC from `/faculty-portal/src/lib/auth.ts`.

## Tasks

<task id="1">
Update `validate_magic_token()` RPC function to call `record_notification_view()` after updating token usage. Add this to the same migration file from 06-01 (`supabase/migrations/20260122_faculty_scheduler_phase6_response_tracking.sql`).

The key insight is that we should record the view BEFORE checking if the instructor is active/valid, so views are recorded even for expired/invalid tokens.

```sql
-- ============================================================================
-- FUNCTION: Validate Magic Token (Updated for Response Tracking)
-- Records notification view on every token validation attempt
-- ============================================================================
CREATE OR REPLACE FUNCTION faculty_scheduler.validate_magic_token(
  p_token TEXT
)
RETURNS TABLE (
  instructor_id UUID,
  first_name TEXT,
  last_name TEXT,
  email TEXT,
  firm_state TEXT,
  tier_designation INTEGER
) AS $$
DECLARE
  v_instructor_id UUID;
BEGIN
  -- Get instructor_id from token first (before any validation)
  SELECT mt.instructor_id INTO v_instructor_id
  FROM faculty_scheduler.magic_tokens mt
  WHERE mt.token = p_token;

  -- Record notification view if we found an instructor
  -- This happens even if token is expired or instructor inactive
  -- (instructor saw the notification, even if they can't act on it)
  IF v_instructor_id IS NOT NULL THEN
    PERFORM faculty_scheduler.record_notification_view(v_instructor_id);
  END IF;

  -- Update usage stats (existing behavior)
  UPDATE faculty_scheduler.magic_tokens
  SET last_used_at = NOW(), use_count = use_count + 1
  WHERE token = p_token;

  -- Return instructor info (existing behavior)
  RETURN QUERY
  SELECT f.id, f.first_name, f.last_name, f.email, f.firm_state, f.tier_designation
  FROM faculty_scheduler.magic_tokens mt
  JOIN faculty f ON f.id = mt.instructor_id
  WHERE mt.token = p_token
    AND f.faculty_status = 'active';
END;
$$ LANGUAGE plpgsql;

COMMENT ON FUNCTION faculty_scheduler.validate_magic_token IS
  'Validates a magic token and returns instructor info. Records notification view on first access (even if token expired). Updates usage stats.';
```
